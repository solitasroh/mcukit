## Executive Summary

| 관점 | 내용 |
|------|------|
| **문제 (Problem)** | 개발자가 Claude Code로 코드를 생성할 때 세션마다 코딩 관습이 리셋되며, 코드 설계 품질(복잡도, SOLID, 네이밍)과 언어별 숙어(C++ RAII, C# async/await)를 체계적으로 검증할 범용 코드 리뷰어가 없다. 도메인 안전 규칙(MISRA, MVVM)은 기존 에이전트가 담당하지만, 그 위의 범용 설계 품질 계층이 부재하다 |
| **해결책 (Solution)** | (1) 범용 코드 리뷰어 + 언어별 리뷰어: `/code-review` 한 번으로 코드 설계 품질(SOLID, 복잡도, DRY) + 언어별 숙어(C/C++, C#, Python)를 검증. 도메인 규칙은 기존 에이전트(safety-auditor, wpf-architect 등)에 위임 (2) 인스팅트 학습 엔진: 프로젝트 코딩 관습을 세션 간 누적 학습하여 반복 교정 제거 |
| **기능-UX 효과 (Function UX Effect)** | 리뷰 시간 70% 단축, 세션 간 코드 일관성 50% 향상, 코드 설계 품질 점수 30% 향상 |
| **핵심 가치 (Core Value)** | "코드 설계 품질은 도메인과 무관하다" -- ECC의 언어별 리뷰어 철학을 계승하면서, rkit의 도메인 안전 에이전트와 계층적으로 결합하여 범용 품질 + 도메인 안전의 이중 보장을 제공한다 |

## 1. 기회 발견 (Opportunity Discovery)

*출처: pm-discovery 에이전트 분석*

### 1.1 목표 결과 (Desired Outcome)

**"세션 간 코드 품질 일관성 50% 향상"**

현재 Claude Code 세션이 종료되면 프로젝트별 코딩 관습(네이밍, 패턴, 도메인 규칙)이 사라지고, 새 세션마다 동일한 교정을 반복해야 한다. 이 문제를 해결하여 일관성 지표를 50% 이상 개선하는 것이 목표다.

## 2. 가치 제안 및 전략 (Value Proposition & Strategy)

*출처: pm-strategy 에이전트 분석*

### 2.1 JTBD 가치 제안 (6-Part)

#### 기능 A: 범용 코드 리뷰어 + 언어별 리뷰어

| 파트 | 내용 |
|------|------|
| **누구 (Who)** | Claude Code를 일상 도구로 사용하는 모든 개발자. 특히 C/C++, C#, Python을 사용하는 개발자 (임베디드에 국한되지 않음) |
| **왜 (Why)** | "코드 설계 품질(SOLID, 복잡도, DRY, 에러 처리)을 도메인과 무관하게 체계적으로 검증받고 싶다. 추가로 사용하는 언어의 숙어(idiom)도 점검받고 싶다" |
| **이전 (What Before)** | rkit의 code-analyzer가 범용 분석을 하지만 언어별 깊이가 얕음. C++ RAII 패턴, C# async/await 안티패턴, Python 타입힌트 누락 같은 언어 고유의 설계 품질은 검사 못함. ECC는 9개 언어별 전문 리뷰어를 보유하여 이 영역을 커버 |
| **어떻게 (How)** | 3-Layer 아키텍처: (1) **L1 범용 설계 품질** - SOLID, 복잡도, DRY, 네이밍, 에러 처리 등 도메인 무관 검증 (2) **L2 언어별 숙어** - C/C++(RAII, const correctness), C#(async/await, nullable), Python(타입힌트, context manager) (3) **L3 도메인 안전** - 기존 에이전트에 위임(safety-auditor, wpf-architect 등), `--domain` 옵션으로 opt-in |
| **이후 (What After)** | `/code-review` 기본 실행으로 L1+L2 통합 리포트. 설계 품질과 언어 숙어를 한눈에 확인. 도메인 규칙은 필요 시 `--domain`으로 추가 |
| **대안 대비 (Alternatives)** | ECC는 9개 언어별 리뷰어를 보유하나 선언적 마크다운 기반(런타임 분석 없음). Copilot Code Review는 범용이나 언어별 깊이 부족. CodeRabbit은 PR 기반 전용. rkit은 metrics-collector.js 정량 분석 + LLM 정성 분석 하이브리드 |

**가치 제안문**: "코드 설계 품질은 도메인과 무관하다 -- `/code-review` 한 번으로 SOLID, 복잡도, 언어 숙어를 검증하고, 도메인 안전은 전문 에이전트에게 맡긴다"

#### 기능 B: 인스팅트 학습

| 파트 | 내용 |
|------|------|
| **누구 (Who)** | 장기 프로젝트를 진행하는 임베디드 개발자/팀 |
| **왜 (Why)** | "프로젝트의 코딩 관습을 AI가 기억하고 반복 교정 없이 일관된 코드를 생성해 주길 원한다" |
| **이전 (What Before)** | 새 Claude Code 세션마다 "우리 프로젝트에서는 이렇게 해"를 반복 설명, CLAUDE.md를 수동으로 유지보수 |
| **어떻게 (How)** | 코드 리뷰 결과 + 사용자 교정 패턴을 자동 수집하여 프로젝트별 인스팅트 프로파일을 누적, 새 세션 시작 시 자동 로드 |
| **이후 (What After)** | 3세션 후부터 반복 교정이 70% 감소, 세션 간 코드 일관성 50% 향상 |
| **대안 대비 (Alternatives)** | GitHub Copilot은 세션 간 학습 없음, ECC는 CLAUDE.md에 의존하되 자동 업데이트 없음 -- rkit 인스팅트는 자동 누적 학습 |

**가치 제안문**: "세션이 끝나도 AI는 당신의 코딩 습관을 잊지 않는다 -- 반복 교정 없는 일관된 코드 생성"

| **채널** | Claude Code 마켓플레이스, GitHub 릴리즈, 임베디드 개발자 커뮤니티 (EEVblog, r/embedded), 기술 블로그 |

### 2.3 전략적 우선순위 결론

**코드 리뷰어 먼저 (v0.9.11-12) --> 인스팅트 다음 (v0.10.x)**

근거 (출처: pm-strategy 분석):
1. **L1 범용 설계 품질 리뷰어**는 metrics-collector.js 등 기존 인프라를 즉시 활용할 수 있어 구현 비용이 낮다. SOLID, 복잡도, DRY 검증은 도메인과 무관하므로 모든 사용자에게 즉시 가치를 제공한다
2. **L2 언어별 숙어 리뷰어**는 ECC의 9개 언어별 리뷰어 접근을 참고하되, rkit 맥락에서 필수인 C/C++, C#, Python 3개 언어부터 시작한다
3. L1+L2 코드 리뷰 결과가 **인스팅트 학습의 데이터 소스**가 되므로, 리뷰어가 먼저 있어야 학습 품질이 높아진다
4. 도메인 안전 규칙(L3)은 기존 에이전트(safety-auditor, mcu-critical-analyzer, wpf-architect)가 이미 충분히 커버하므로 새로 만들 필요 없이 **오케스트레이션만 추가**한다

## 5. 제품 요구사항 (PRD)

### 5.1 요약

rkit에 ECC 인사이트를 통합하여 두 가지 핵심 기능을 구현한다.

첫째, **3-Layer 코드 리뷰 시스템**:
- **Layer 1 (범용 설계 품질)**: SOLID 원칙, 순환 복잡도, DRY, 네이밍, 에러 처리 등 **도메인과 무관한** 코드 설계 품질 검증. C든 C#이든 Python이든 동일하게 적용.
- **Layer 2 (언어별 숙어)**: C/C++ 숙어(RAII, const correctness, smart pointer), C#(async/await, nullable, dispose), Python(타입힌트, context manager) 등 **언어 고유 설계 패턴** 검증. ECC의 9개 언어별 리뷰어 접근을 참고.
- **Layer 3 (도메인 안전 규칙)**: MISRA C(safety-auditor), ISR/DMA(mcu-critical-analyzer), MVVM(wpf-architect) 등은 **기존 전문 에이전트에 위임**. `--domain` 옵션으로 opt-in.

핵심 철학: **코드 설계 품질은 도메인 영역과 무관하다.**

둘째, **인스팅트 학습 엔진**은 L1+L2 코드 리뷰 결과와 사용자 교정 패턴을 세션 간 누적하여 반복 교정 없이 일관된 코드를 생성한다.

### 5.2 배경 및 맥락

**왜 지금인가?**

1. **Claude Code 시장 지배**: 2026년 AI 코딩 도구 선호도 조사에서 Claude Code가 46%로 1위 (출처: pm-research). 플랫폼 위의 전문 플러그인 수요가 급증
2. **ECC의 성공이 증명한 수요**: ECC가 146K 스타를 달성한 핵심 요인 중 하나가 9개 언어별 전문 리뷰어. 코드 설계 품질에 대한 체계적 검증 수요가 명확히 존재
3. **rkit 인프라 성숙**: metrics-collector.js(C/C++/C#/JS/TS/Python 정량 분석), 기존 도메인 에이전트(safety-auditor, wpf-architect 등) 41개가 이미 구축되어 3-Layer 아키텍처의 기술적 기반이 충분
4. **범용→도메인 계층 분리의 가치**: 코드 설계 품질(L1+L2)은 모든 개발자에게 가치를 제공하고, 도메인 안전(L3)은 임베디드 개발자에게 추가 가치를 제공. 시장을 넓히면서도 전문성을 유지하는 전략

**무엇이 가능해졌나?**

- rkit v0.9.x에서 metrics-collector.js, PDCA quality gate, 기존 도메인 에이전트가 안정화되어 3-Layer 코드 리뷰의 기술적 기반이 완성됨
- ECC 분석을 통해 언어별 리뷰어의 구체적 패턴(cpp-reviewer, csharp-reviewer 등)을 확보
- Claude Code의 세션 간 상태 저장 메커니즘(CLAUDE.md, .rkit/)이 인스팅트 학습의 저장소로 활용 가능

### 5.7 가정 및 리스크

| A2 | 언어별 숙어 리뷰(L2)가 범용 리뷰(L1) 대비 추가 가치를 제공한다 | Value | 중 | A/B 비교: L1만 vs L1+L2 적용 시 발견 이슈 수 비교 (E2) |

| A5 | L1+L2 코드 리뷰 결과가 인스팅트 학습의 충분한 데이터 소스가 된다 | Feasibility | 중 | 데이터 분석 |

**주요 리스크**:
- **R1**: 인스팅트 학습 수렴이 예상보다 느릴 수 있음 --> 완화: 초기에는 rule-based fallback 제공
- **R2**: Claude API 비용이 리뷰 빈도에 비례하여 증가 --> 완화: 캐싱 전략 + diff 기반 증분 리뷰
- **R3**: L2(언어 숙어) 리뷰어의 false positive가 높을 수 있음 --> 완화: 언어별 20파일 검증 후 규칙 조정
